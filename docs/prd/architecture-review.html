Two adapters justify the seam: file+HTTP in production, in-memory in tests.

Problem. The resolver does file discovery, HTTP fetching, parsing and resolution. The interesting logic can only be reached through the filesystem — or by reflection on private methods.

Solution. Inject an IExternalDocumentSource that maps an external-resource string to a parsed document. The resolver keeps the logic; transport moves behind the seam.

  • ▪ the interface becomes the test surface
  • ▪ two adapters justify a real seam
  • ▪ resolution tested in-memory — no temp files
  • ▪ delete reflection-on-private-method tests
  • ▪ I/O stops leaking into the walk

04Localize the Microsoft.OpenApi reflection

Speculative single adapter

OpenApiDocumentExtensions.cs — OpenApiReferenceExtensions

Before — interface as wide as implementation

3 helpers,
thin bodies

Reflection on init-only Reference props by string name. Fragile against the library — but already in one place.

After — one deep adapter over the library

small interface,
reflection hidden

One OpenApiReferenceGateway hides the reflection; callers see typed reads/writes.

Problem. Init-only Reference properties are read and written by reflection with string property names — fragile against Microsoft.OpenApi upgrades.

Solution. Keep it in one deep adapter with a typed interface, so the reflection is the implementation and never the caller's concern.

Honest read: only one adapter exists (the real library). One adapter = a hypothetical seam. No port is justified — this is localization, not a new seam.
  • ▪ reflection concentrates in one module
  • ▪ interface shrinks; implementation absorbs it
  • ▪ low urgency — already mostly localized

Top recommendation

Start with 02 — put a seam under document loading.

It buys the largest testability win and unblocks the rest. Today the resolution logic is only reachable through disk, network, or reflection on private methods — the clearest sign the interface is the wrong shape. A document-source seam with two adapters (file+HTTP in production, in-memory in tests) makes the interface the test surface, and gives card 03 the dependency it needs to collapse the two passes into one deep merge module. Card 01 stands alone and is the cheapest quick win.
